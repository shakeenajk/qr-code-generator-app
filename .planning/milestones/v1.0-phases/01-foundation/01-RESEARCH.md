## Summary

This phase establishes the Astro 5 project from scratch: scaffolding, Tailwind v4 via the Vite plugin, React integration for future islands, and all SEO instrumentation before any generator code is written. The stack is fully decided — Astro 5 (currently v5.18, stable), Tailwind v4 (stable since January 2025, Vite-plugin-based), `@astrojs/react`, `@astrojs/sitemap`, deployed to Vercel with zero adapter configuration for static output.

SEO requirements drive most of the work: meta tags and Open Graph in the Astro `<head>`, two JSON-LD schemas (`WebApplication` and `FAQPage`) injected via Astro's `set:html` directive, a sitemap generated by the official integration, and a `robots.txt` served as a static file from `public/`. The SVG logo is hand-authored inline SVG — a grid of small rectangles (`<rect>`) masked to the silhouette of the letter "Q", rendered in the brand blue.

Phase 1 is the structural skeleton that Phase 2 drops the QR generator island into; the generator area must be stubbed as an empty placeholder `<div>` with a predictable ID or slot so Phase 2 requires no layout surgery.

**Primary recommendation:** Scaffold with `npm create astro@latest`, add React + Tailwind v4 + sitemap integrations immediately, then build the Layout component with all head meta and JSON-LD in one pass before writing any page content.

### Pattern 1: Layout Component Owns All Head Meta

**What:** One `Layout.astro` accepts `title`, `description`, `ogImage` props and emits all `<head>` content including meta tags, OG tags, and both JSON-LD `<script>` blocks.

**When to use:** Every page. For this project there is one page, so the Layout also hardcodes the schemas directly rather than passing them as props.

**Example:**
```astro
---
// src/layouts/Layout.astro
interface Props {
  title: string;
  description: string;
  ogImage?: string;
}
const { title, description, ogImage = '/og-image.png' } = Astro.props;
const canonicalUrl = new URL(Astro.url.pathname, Astro.site);

const webAppSchema = {
  "@context": "https://schema.org",
  "@type": "WebApplication",
  "name": "QRCraft",
  "url": Astro.site?.toString(),
  "description": "Free online QR code generator. Create custom QR codes for URLs, text, WiFi, and contact cards — no signup required.",
  "applicationCategory": "UtilitiesApplication",
  "operatingSystem": "All",
  "browserRequirements": "Requires JavaScript.",
  "offers": {
    "@type": "Offer",
    "price": "0",
    "priceCurrency": "USD"
  }
};

const faqSchema = {
  "@context": "https://schema.org",
  "@type": "FAQPage",
  "mainEntity": [
    {
      "@type": "Question",
      "name": "How do I create a QR code for a website?",
      "acceptedAnswer": {
        "@type": "Answer",
        "text": "Paste your URL into QRCraft, customize colors and style, then download as PNG or SVG."
      }
    }
    // ... additional Q&A pairs matching visible FAQ section
  ]
};
---
<html lang="en">
  <head>
    <meta charset="UTF-8" />
    <meta name="viewport" content="width=device-width, initial-scale=1" />
    <title>{title}</title>
    <meta name="description" content={description} />
    <link rel="canonical" href={canonicalUrl} />
    <!-- Open Graph -->
    <meta property="og:title" content={title} />
    <meta property="og:description" content={description} />
    <meta property="og:image" content={new URL(ogImage, Astro.site)} />
    <meta property="og:type" content="website" />
    <meta property="og:url" content={canonicalUrl} />
    <!-- JSON-LD -->
    <script type="application/ld+json" set:html={JSON.stringify(webAppSchema)} />
    <script type="application/ld+json" set:html={JSON.stringify(faqSchema)} />
    <slot name="head" />
  </head>
  <body>
    <slot />
  </body>
</html>
```

### Pattern 2: JSON-LD via `set:html`

**What:** Astro's `set:html` directive bypasses HTML escaping, which is required for `<script>` tags containing JSON. Without `set:html`, Astro escapes `<`, `>`, and `&` inside script content, producing invalid JSON-LD.

**When to use:** Any time you emit a `<script type="application/ld+json">` block in Astro. Always use this — never interpolate JSON directly without `set:html`.

**Example:**
```astro
<script type="application/ld+json" set:html={JSON.stringify(schema)} />
```

**Source:** Verified against Astro GitHub issue #3544 and community patterns. The `set:html` approach is the authoritative solution.

### Pattern 3: Sitemap + robots.txt

**What:** `@astrojs/sitemap` auto-discovers all static routes and generates `sitemap-index.xml` (index) + `sitemap-0.xml` (entries). A static `public/robots.txt` references the sitemap.

**Required config in `astro.config.mjs`:**
```javascript
import { defineConfig } from 'astro/config';
import sitemap from '@astrojs/sitemap';
import tailwindcss from '@tailwindcss/vite';
import react from '@astrojs/react';

export default defineConfig({
  site: 'https://qrcraft.app',   // REQUIRED for sitemap generation
  integrations: [react(), sitemap()],
  vite: {
    plugins: [tailwindcss()],
  },
});
```

**`public/robots.txt` content:**
```
User-agent: *
Allow: /

Sitemap: https://qrcraft.app/sitemap-index.xml
```

### Pattern 5: SVG Logo Construction

**What:** The QRCraft logo renders the letter "Q" silhouette built from small square dots arranged on a grid, mirroring the visual language of a QR code. Implemented as inline SVG for CSS color control.

**Approach:** Draw a 7×7 grid of `<rect>` elements, each 10×10 px with 2px gap. Selectively include rects that approximate the letter "Q" outline (thick ring shape with a tail). Color all rects with `fill="currentColor"` so the parent `class="text-blue-600"` drives the color.

**Example skeleton:**
```astro
<!-- src/components/Logo.astro -->
<svg
  xmlns="http://www.w3.org/2000/svg"
  viewBox="0 0 84 84"
  width="40"
  height="40"
  aria-label="QRCraft logo"
  role="img"
  class="text-[#2563EB]"
>
  <!-- Each rect: x = col*12, y = row*12, width=10, height=10, fill="currentColor" -->
  <!-- Include rects for rows/cols that form the letter Q outline -->
  <rect x="24" y="0"  width="10" height="10" fill="currentColor" />
  <!-- ... remaining dots for Q shape ... -->
</svg>
```

**Calibration tip:** Map the letter "Q" on a 7-column grid: columns 2–5 in row 0 (top bar), columns 1 and 5 for side bars in rows 1–4, columns 2–5 in row 5 (bottom bar), column 4 row 5 + column 5 row 6 (the tail).

### Anti-Patterns to Avoid

- **Using `@astrojs/tailwind` with Tailwind v4:** This integration is deprecated for v4. Use `@tailwindcss/vite` instead (directly in `vite.plugins`).
- **Omitting `site:` from `astro.config.mjs`:** The sitemap integration silently produces no output if `site` is not set.
- **Interpolating JSON into `<script>` without `set:html`:** Astro escapes special chars, breaking JSON-LD parsing.
- **Putting `robots.txt` in `src/pages/` as TypeScript endpoint:** Unnecessary complexity for this single-domain static site; static file in `public/` is idiomatic and simpler.
- **Multiple `<h1>` tags on the page:** SEO violation. One `<h1>` ("Free QR Code Generator" or equivalent), all subsequent headings `<h2>`.
- **Placing JSON-LD schemas in `<body>` outside `<head>`:** While technically valid per spec, Google recommends `<head>` placement. Always put JSON-LD in Layout's `<head>`.
- **FAQPage schema content diverging from visible FAQ text:** Google validates that schema content matches visible page content. Keep them in sync — ideally derive both from the same data structure.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Sitemap generation | Custom sitemap builder script | `@astrojs/sitemap` | Handles URL discovery, lastmod, priority, chunking automatically |

**Key insight:** Astro's official integration CLI (`npx astro add X`) is idempotent and handles all config mutations safely. Use it for every integration rather than manual config edits to avoid stale peer deps or missed tsconfig changes.

## Common Pitfalls

### Pitfall 1: `site` Missing from Astro Config

**What goes wrong:** `@astrojs/sitemap` generates no output (or outputs empty sitemap) at build time, and `Astro.site` is `undefined` in layouts — breaking canonical URL and OG image absolute URL construction.

**Why it happens:** The `site` property is optional in Astro config for development, so the build succeeds but the sitemap is empty.

**How to avoid:** Set `site: 'https://yourdomain.com'` in `astro.config.mjs` from day one, before adding the sitemap integration. Verify the output `dist/sitemap-index.xml` exists after first build.

**Warning signs:** `Astro.site` is `undefined` in a component; `dist/` has no sitemap files after build.

### Pitfall 2: Tailwind v4 Class Detection Issues

**What goes wrong:** Tailwind v4's automatic content detection scans `.astro`, `.html`, `.js`, `.ts`, `.jsx`, `.tsx` files. Classes defined in dynamically assembled strings (e.g., template literals or object maps) may not be detected and will be purged.

**Why it happens:** Tailwind v4 uses static regex scanning; it cannot evaluate runtime expressions.

**How to avoid:** Always write full class names as string literals in markup. Use `safelist` in a CSS `@layer` comment or `@source` directive for any dynamic classes. Avoid string concatenation for class names.

**Warning signs:** A class works in dev but disappears in production build.

### Pitfall 3: FAQPage Schema Out of Sync with Visible Content

**What goes wrong:** Google's Rich Results Test flags mismatches between JSON-LD FAQ content and visible page text, potentially suppressing FAQ rich snippets.

**Why it happens:** Schema is edited separately from the visible FAQ component, and they drift.

**How to avoid:** Define FAQ Q&A pairs as a TypeScript array in the Layout or a shared data file. Pass it to both the visible `<FAQ>` component and the JSON-LD schema builder. Single source of truth.

**Warning signs:** Google Search Console shows "FAQ structured data items do not match visible text."

### Pitfall 4: Missing `set:html` on JSON-LD Script Tags

**What goes wrong:** Astro's default HTML escaping converts `"` to `&quot;`, `<` to `&lt;`, etc., producing malformed JSON that schema validators reject.

**Why it happens:** Astro's template sanitization applies to all interpolated values including script content, for XSS prevention.

**How to avoid:** Always use `<script type="application/ld+json" set:html={JSON.stringify(schema)} />`. Never use `{JSON.stringify(schema)}` inside script tags without `set:html`.

**Warning signs:** Google Rich Results Test shows "Invalid JSON" error; browser console shows JSON parse errors.

### Pitfall 5: Semantic HTML Heading Hierarchy Violations

**What goes wrong:** Multiple `<h1>` elements, or skipping heading levels (e.g., `<h1>` then `<h3>` with no `<h2>`), reduce SEO signal and break accessibility.

**Why it happens:** Developers style headings by visual appearance rather than semantic role, using `<h3>` when they want smaller text.

**How to avoid:** One `<h1>` per page (the primary keyword phrase). Use `<h2>` for all major sections (Features, FAQ, etc.). Size headings with Tailwind classes, not by choosing a lower heading level.
